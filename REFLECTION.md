Reflection Questions: 

1. Walk us through the code: 
I structured the dashboard around a Table-first design to prioritize the high-frequency actions of sorting, searching, and filtering. Choosing a contextual modal rather than a separate 'Create' page allows users to input applicant data instantly and see real-time score updates without losing their place in the list. To ensure a cohesive experience, I mirrored this table layout on the Applicant Detail page, providing a transparent breakdown of individual scores and their weighted contributions. Finally, the Summary page utilizes a card-based layout to highlight outliers, such as top and bottom performers, transforming raw data into manageable, high-level statistics.

2. What did Claude Code get wrong? 
For the most part, Claude's logic was sound, but it struggled with specific styling persistence. A notable example occurred while styling the dashboard table: the header column names appeared to be missing. Claude insisted the issue was rooted in a Tailwind configuration error, but upon manual inspection, I discovered it had applied a text-white attribute to a white header, rendering the text invisible. Even after I corrected it, Claude repeatedly attempted to re-insert that specific tag in subsequent iterations.

3. What would you improve with more time?
With more time, I would focus on two key areas: design iteration and data persistence. While I’m satisfied with the current dashboard layout, I’d like to experiment with more advanced data visualization techniques to make the trends even more accessible. More importantly, I would evolve the 'Add Applicant' flow from a temporary client-side session to a fully persistent backend. Currently, new entries exist only in the local browser state, which is why the Detail pages—which rely on unique identifiers—don't yet recognize them. Additionally, I would implement stricter coding standards by introducing comprehensive TypeScript interfaces and refactoring the component logic to better group methods and functionality. This would not only improve the application's stability but also ensure the codebase is scalable and easily navigable for a larger development team.

4. How was the experience of working with an unfamiliar framework + AI tooling?
Utilizing Nuxt 3 with the assistance of Claude was a highly productive experience. I found that while the architectural layout differed from frameworks I’ve used previously, the underlying logic was similar enough that I could quickly find my rhythm. Claude acted as an effective bridge during this process; whenever I encountered a Nuxt-specific convention that was unfamiliar, I used the AI to get a conceptual baseline and then supplemented that with my own research. This hybrid approach allowed me to bypass the initial 'blank page' frustration of a new framework and focus on building out the dashboard logic.
